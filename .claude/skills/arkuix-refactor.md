# Skill: /arkuix-refactor

Four-round refactoring of ArkUI / ArkUI-X ArkTS code.
Works on **any** `.ets` file — generated by `/generate`, converted by `/kmp-to-arkuix`, or pre-existing.

> **执行前必须**遵守 [`AGENTS.md`](../../AGENTS.md) 的"通用前置协议（Git 状态检查）"。

```
Round 1: scan      (LLM auto)
Round 2: execute   (User selects → LLM applies Edit)
Round 3: verify    (LLM runs check-api-parity.sh)
Round 4: distill   (LLM drafts rule → User confirms → append to feedback/refactor-rules.md)
```

## Trigger

User types `/arkuix-refactor` optionally followed by:
- `<file-path>` — target `.ets` file (defaults to last mentioned / most recently generated file)
- `--goal <performance|memory|readability|api-upgrade>` — focus area (defaults to `performance`)

## Required Reference

每轮扫描前必读：`.claude/references/arkuix-refactor-checklist.md`（完整检查清单 + issue ID 体系 + 输出格式）。

## Responsibility Split

| Step | Who | What |
|---|---|---|
| Provide file path | **User** | 输入 `/arkuix-refactor path/to/File.ets` |
| Choose focus goal | **User** (可选) | `--goal performance` 等；默认 performance |
| Round 1 scan | **LLM** | 读文件 + Read checklist → 输出编号 issue 列表 |
| Round 2 prioritise | **User** | 输入 issue 编号（如 `1 3 5` / `P1 P3` / `all` / `skip`） |
| Round 2 apply | **LLM** | Edit 应用所选修复 |
| Round 3 verify | **LLM** | 跑 `scripts/check-api-parity.sh`，报告 PASS/WARN/FAIL |
| Round 4 distill | **LLM proposes, User confirms** | 起草可复用规则 → 用户 yes/no → 追加到 `feedback/refactor-rules.md` |
| Stop or new round | **User** | 输入 `done` 或开新一轮 |

## Round 0 — Preflight（每次调用首步，强制）

```bash
bash scripts/preflight.sh
```

输出 git 状态 / 一致性 / 残留追踪。如 git 非干净 → 按 [`AGENTS.md`](../../AGENTS.md) 规则 13 询问用户。

## Round Protocol（详细）

### Round 1 — Scan

```
1. Read target file
2. Read .claude/references/arkuix-refactor-checklist.md (首轮加载，后续轮复用)
3. 按 checklist + --goal 筛选 → 收集所有匹配 issue
4. 输出编号列表（含行号 + issue ID + 修复建议）
5. 等待用户输入 (e.g. "1 3 5" / "P1 P3" / "all" / "skip")
```

### Round 2 — Execute

```
1. 对每个用户选中的 issue：
   - 用 Edit 工具应用一处修改（不用 Write 整体覆盖）
   - 修改后重读文件验证未破坏结构
2. 报告每处变更（before / after diff）
```

### Round 3 — Verify

```
1. 跑 bash scripts/check-api-parity.sh <修改的 ets 文件>
2. 解析输出：
   - PASS → 进入 Round 4
   - WARN → 列出 warning 项，问用户是否处理（处理则回到 Round 2）
   - FAIL → 必须立即回到 Round 2 修复
```

### Round 4 — Distill（沉淀可复用规则）

仅当本轮修复包含**可机械化检测**的模式时执行：

```
1. LLM 起草一条规则，格式：

   ---
   ### RULE-NN [日期]：[规则名]

   - **来源**：[文件名] Round X 重构
   - **问题模式**：[grep / 描述]
   - **修复模式**：[替换为...]
   - **检测命令**：`grep -rn "<pattern>" <path>` 或 `bash scripts/check-api-parity.sh <path>`
   - **优先级**：P0/P1/P2
   ---

2. 询问用户：「这条规则要追加到 feedback/refactor-rules.md 吗？(yes/no/edit)」

3. 用户确认后用 Edit 追加到 feedback/refactor-rules.md 末尾

4. 如果规则适合机械化（grep 可表达），提示用户：
   「该规则可加入 scripts/check-api-parity.sh，需要我起草脚本片段吗？」
```

## Round 4.5 — Feature Log（强制，每次重构完成后）

重构完成后**必须**追加 feature log（一次 `/arkuix-refactor` 调用 = 一条日志）：

```bash
# 推断：ets 文件名 PascalCase → kebab-case
bash scripts/new-feature-log.sh <inferred-name> refactor-pass-<round-id>
```

agent 直接撰写 4 段：
- 本轮目标：重构焦点（性能/内存/可读性/API升级）
- 改动要点：本轮选择的 issue 编号 + Edit 应用细节
- 验证结果：check-api-parity 输出 + Round 4 distill 起草规则（如有）
- 残留：未选择修复的 issue 编号

## Round 5 — Classify & Notify（强制，本轮所有 Edit 完成后）

```bash
bash scripts/classify-change.sh
```

如退出码 ≠ 0 → 必须把 `🔔 元变更检测` 醒目块原样复述给用户。重构通常是业务路径（`arkuix-workspace/` 或 `benchmarks/`），多为 business；若同时改了 mapping/checklist 等元文件则为 meta/mixed。

## Rules

- 每次只用 Edit 修一处（不用 Write 整体覆盖）
- 应用后重读文件验证未破坏结构
- 不给非页面组件加 `@Entry`
- 添加 `IDataSource` 时放在使用它的 `@Component` 之上、同文件内
- 抽 `@Builder` 时放在 `@Component struct` 内
- Round 3 的 check-api-parity.sh FAIL 必须修复才能进入 Round 4
- Round 4 由 LLM 起草，**必须**等用户确认再追加（不得静默 append 到 refactor-rules.md）
