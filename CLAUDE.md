## 目录结构速览

> 使用者按实际工程结构填入。模板默认提供以下框架层目录，业务目录自行添加。

```
YourProject/
├── .claude/
│   ├── skills/           ← 所有 Skill 定义（触发 AI 工作流）
│   └── references/       ← API 映射表、最佳实践参考（按需 Read）
├── scripts/              ← 元框架脚本 + 项目自定义脚本
├── feedback/
│   ├── meta/             ← 元变更归档（规则/接口/流程，带编号）
│   └── features/         ← 业务功能演化档案（按特性归档）
├── stats/                ← Agent 提交 token 统计
└── reports/              ← 输出报告（/run-benchmark 等自动生成）
```

## 机械化校验

`scripts/check-api-parity.sh` 对项目代码做硬性合规检查，由相关 Skill 自动调用：

```bash
bash scripts/check-api-parity.sh <file-or-dir>
# 退出码：0 PASS / 1 FAIL / 2 WARN
```

**check-api-parity.sh（项目代码合规规则）**

> 使用者在 `scripts/check-api-parity.sh` 中按项目添加具体规则，并在此表同步更新。
> 规则 ID 命名约定：`P-*` 性能 / `R-*` 资源泄漏 / `S-*` 状态管理 / `C-*` 代码复杂度。

| 规则 ID | 检查 | 等级 |
|---|---|---|
| （使用者自定义） | — | — |

**check-consistency.sh（元数据 + 结构一致性，20 条）**

| 规则 ID | 检查 | 等级 |
|---|---|---|
| M-SKILL-01 | skill 文件数与 CLAUDE.md 速查表行数一致 | FAIL |
| M-MAP-01 | mapping 文件数与 CLAUDE.md 列表一致 | FAIL |
| M-RULE-01 | parity 规则数与 CLAUDE.md 表行数一致 | WARN |
| M-FB-01 | feedback 编号连续 1..N | FAIL |
| M-AGENTS-01 | 所有顶层子目录均有 AGENTS.md | WARN |
| M-ROOT-01 | 根 AGENTS.md 覆盖所有主要子目录 | WARN |
| M-FB-FORMAT | feedback（N≥4）必含 5 段必备标题 | FAIL |
| M-MAP-AP | mapping-*.md 必含 `## Anti-Patterns` 节 | FAIL |
| M-SKILL-PREFLIGHT | skill 必含 AGENTS.md + Git 前置引用 | FAIL |

| M-SKILL-SUMMARY-CLASSIFY | skill 必含 classify-change.sh 引用 | FAIL |

规则背后的"为什么"见 [`feedback/DESIGN.md`](feedback/DESIGN.md)。

**preflight、classify、feedback 生成器**：

```bash
bash scripts/preflight.sh                          # skill 入口仪式（git 状态 + 改动分类 + 一致性 + 残留追踪）
bash scripts/classify-change.sh                    # 分类 staged 改动（meta / business / mixed）；meta 时输出醒目块
bash scripts/new-feedback.sh <slug>                # 新建本轮 feedback（自动编号 + 5 段模板，写入 feedback/meta/）
bash scripts/query-pending.sh                      # 查询归档中未解决的 - [ ] 残留项
```

**元变更 vs 业务迭代**：[`feedback/AGENTS.md`](feedback/AGENTS.md) 定义两类边界。元变更必须随同 feedback 提交，pre-commit hook 强制；业务迭代不需。

**归档目录结构**：`feedback/` 是 agent 交互归档的总入口，分两类：
- `feedback/meta/` —— 元变更归档（规则/接口/流程/Skill 变更），带连续编号
- `feedback/features/` —— 业务功能演化档案（按特性归档，多轮迭代）

AI 驱动的业务改动（skill 触发）必须自动留 feature log（agent 自动调用 `bash scripts/new-feature.sh` / `new-feature-log.sh` 并填内容）。`/run-benchmark` 例外（只读豁免）。

**残留项强制规则**：每轮 feedback（无论 meta 或 feature log）必须在"残留/下一轮"节用 `- [ ]` / `- [x]` 复选框标记未解决/已解决项。Agent 每轮开始时必须用 `bash scripts/query-pending.sh` 查询上轮残留，有未解决项时需告知用户。M-PENDING-01 规则在 check-consistency.sh 中扫描并 WARN。

**Token / 改动量统计**：[`stats/`](stats/AGENTS.md) 记录每次 **agent 提交**的改动量与 token 消耗；用户手动提交不记录。Agent 在每次 `git commit` 后必须主动调用 `bash scripts/log-tokens.sh --from-commit HEAD`。Token 字段需用户从 UI 读后用 `bash scripts/log-tokens.sh annotate <round> --input-tokens N --output-tokens N` 回填。统计分析用 `bash scripts/stats-report.sh [--by-round|--by-feature|--trend|--top N]`。

**档案绝对不可删**：`feedback/meta/[N]-*.md`、`feedback/features/*/[N]-*.md`、`feedback/features/*/README.md`、`feedback/DESIGN.md`、`feedback/refactor-rules.md` 禁止删除/重命名。Pre-commit（M-FB-01）+ commit-msg（check-archive-deletion）双重拦截。**无 override 通道**——确需清理（极端）只能 `git commit --no-verify` 显式破坏。
