## Plan

> 必备节（FAIL 级 M-FEATURE-PLAN）

### 关键设计

**1. Tantivy 版本与 feature 隔离**

- `tantivy = "0.22"`（2024-2025 稳定主线）
- `arkui-rag-storage/src/tantivy_bm25.rs` 全文 `#[cfg(feature = "tantivy")] pub mod tantivy_bm25;`
- 默认 `cargo check --workspace` 不拉 tantivy（依赖 ~50MB + 编译 ~1-2 分钟）
- 启用：`cargo check -p arkui-rag-storage --features tantivy` / `make check-tantivy`

**2. Schema 设计（自包含）**

| 字段 | 类型 | 用途 |
|---|---|---|
| `id` | STRING + STORED | 精确查找 + upsert 删旧 |
| `content` | TEXT (ngram tokenizer) + STORED | BM25 主体 |
| `heading_path` | TEXT (ngram) + STORED | 多字段加权 / heading boost |
| `source` / `platforms` / `api_version` / `chunk_type` / `tags` | STRING + STORED + INDEXED | 元数据过滤 |
| `deprecated` | BOOL + INDEXED + STORED + FAST | 默认排除 deprecated |
| `meta_json` | TEXT + STORED | 完整 `ChunkMetadata` 的 JSON 序列化 → search 返回完整 Hit |

`meta_json` 设计要点：避免逐字段 deserialize ChunkMetadata（代码量大 + 与 schema 演进耦合），统一 JSON 一步到位。

**3. 中文分词：ngram(2,3) 简化方案**

Tantivy 默认 tokenizer 对中文无效（按空格切，中文文本被当成一个 token）。Day 4 用 **`NgramTokenizer::new(2, 3, false)`** —— 2-gram / 3-gram 切，加 `LowerCaser` + `RemoveLongFilter(40)` 过滤。这是简化方案：

- ✅ 中文能匹配（"下拉刷新" 的 2-gram = "下拉" "拉刷" "刷新"，与 doc 多个 chunk 重叠）
- ✅ 中英混合稳健
- ❌ 精度低于 jieba 分词（如"下拉"和"上拉"会因 1 个 char 不同被认为很相似）
- ⏳ 生产建议：feature `tantivy-jieba` 引入分词器（Day 5+ backlog）

**4. CLI 接口：`--bm25 memory|tantivy`**

| 参数 | 行为 |
|---|---|
| `--bm25 memory`（默认） | 用 `InMemoryBM25Index`（空 stub），RRF 退化为纯向量 |
| `--bm25 tantivy` | 用 `TantivyBM25Index`；需 `--features tantivy` 重新编译，否则 `bail!` 友好报错 |

BM25 目录自动推导：`<index-path-dir>/bm25/`（与 `index.json` 并列）。无需用户额外指定。

**5. 防错配**：当前 `index.json` 只校验 `embedder_model_id`，**不**校验 bm25 后端类型 —— 不严重（BM25 目录存在 / 不存在自然报错）。完整方案 Day 5 可加 `bm25_kind` 字段到 IndexSnapshot。

### 替代方案权衡

- 备选 tantivy-jieba：被否，多一个第三方依赖；ngram 在 demo 阶段够用
- 备选 Lucene-style 复杂 schema（per-language 字段）：被否，单字段 + ngram 够用，避免索引膨胀
- 备选直接 LanceDB 而非 Tantivy 做 BM25：被否，LanceDB 主打向量，BM25 不是它强项；Tantivy 才是 Rust BM25 标准
- 备选 SQLite FTS5：被否，依赖 sqlite native 库，且 BM25 精度不如 Tantivy

### 与 Day 3 OnnxEmbedder 的关系

两个 feature 完全独立：
- `--features onnx` → 真实向量
- `--features tantivy` → 真实 BM25
- `--features full` → 一键开两个
- 用户可自由组合：Mock + Tantivy / OnnxEmbedder + InMemoryBM25 / OnnxEmbedder + Tantivy

混合检索的 4 种状态：
| 向量 | BM25 | 检索质量 |
|---|---|---|
| Mock | Memory | Day 2 demo 级（无语义无关键词） |
| Mock | Tantivy | 关键词匹配可用，语义空 |
| Onnx | Memory | 语义可用，关键词依赖向量 |
| Onnx | Tantivy | **真正的 Hybrid（业界标配）** |

## 改动要点

> API 选型 / 算法 / 关键决策

**与 Day 3 的差异**：
- crate 数仍 8（Tantivy 不新增 crate，只是 storage 加 feature + module）
- 测试数：24 → 31（TantivyBM25Index 加 7 单测：basic/upsert-overwrite/delete/filter-platform/deprecated-default/empty-query/reopen-persist）
- CLI 参数：`--bm25 memory|tantivy`（默认 memory，向后兼容）
- Makefile 新增 build-tantivy / build-full / check-tantivy 三个 target
- workspace deps 加 `tantivy = "0.22"`

**API 选型**：
- `Term::from_field_text` / `Term::from_field_bool` —— Tantivy 0.22 推荐入口
- `QueryParser::set_conjunction_by_default()` —— 多 token 走 AND（"下拉 刷新" → 必须都命中），提高精度
- `TantivyDocument::default()` + `add_text` —— 0.22 新 API（替代旧 `Document::new()`）
- `BooleanQuery::new(clauses)` —— 多条件组合（main + filter）

**算法**：
- ngram(2,3) 中文分词 + L2 norm 配合 vector 路径，最后 RRF k=60 融合
- search 前 `reader.reload()` 强制读取最近 commit（防止索引刚 commit 但 reader 还没刷新）
- delete + add 的 upsert 模式（按 id Term 删旧 → 加新）
