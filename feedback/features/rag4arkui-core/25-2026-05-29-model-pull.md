### 重构策略：抽 download_and_extract

Day 21 `cmd_corpus_pull` 内部混了 3 段逻辑：
1. 目标目录预检
2. 下载 / 读本地（HTTP + tar bytes）
3. 解压 + path traversal 检查

Day 21b 需要复用 (1)+(2)+(3) 但用不同的「成功消息」和「default URL/target 路由」。

抽出共用：
```rust
struct ExtractStats { bytes_mb: f64, entries_count: usize }

async fn download_and_extract(
    url: &str, from_file: Option<&Path>, target: &Path,
    force: bool, strip_components: usize

```

`cmd_corpus_pull` 和 `cmd_corpus_model_pull` 都变成 thin wrappers：
- 解析自己的 URL / target 默认值
- 调 `download_and_extract`
- 打自己的成功消息

### 模型名 → URL 路由（hard-coded 表）

```rust
fn default_model_url(name: &str) -> Result<&'static str> {
    match name {
        "bge-m3" => Ok(BGE_M3_URL),
        "bge-reranker-v2-m3" => Ok(BGE_RERANKER_URL),
        _ => bail!("未知模型名: {} · 已知 bge-m3/bge-reranker-v2-m3 · 或 --url 自定义", name),
    }
}
```

未来加新模型只改这一处。

### 默认 target：`~/.arkui-rag/models/<name>/`

```rust
fn default_model_target(name: &str) -> Result<PathBuf> {
    let home = std::env::var("HOME")
        .or_else(|_| std::env::var("USERPROFILE"))  // Windows 兼容
        .context("HOME / USERPROFILE 都未设置 · 用 --target 指定")?;
    Ok(PathBuf::from(home).join(".arkui-rag").join("models").join(name))
}
```

不引入 `dirs` crate · stdlib 已够。

### HTTP 上限提升到 2 GB

Day 21 corpus 用 500 MB · 模型可能更大（BGE-M3 ONNX ~600 MB · fp32 ~1.2 GB）。
HTTP 上限提到 2 GB · 超时延长到 300s（vs corpus 的 180s）。

### 替代方案权衡（被否）

- 备选 1：用 `git lfs clone` shell out 到 HuggingFace
  - 否决：跨平台脆弱（Windows 无 git-lfs 默认）· 用户要先装 git-lfs · ureq + tarball 更简单
- 备选 2：每个文件单独 HTTP 下载（model.onnx / tokenizer.json / config.json）
  - 否决：要硬编码每个文件路径 · tarball 抽象更干净 + 用户可自打包私有模型
- 备选 3：用 `reqwest` 替代 `ureq`（Day 21 已经选择 ureq · 复用）
  - 否决：违反 Day 21 选型 · 体积爆涨
- 备选 4：用 `dirs` crate 取 home dir
  - 否决：1 个新 dep 不值得 · stdlib 的 HOME/USERPROFILE env var 已足
