## Plan

> 必备节（FAIL 级 M-FEATURE-PLAN）

### 设计：手撸 JSON-RPC 不引入 SDK

**理由**：
- MCP 协议本身简单（JSON-RPC 2.0 + 4 个 method）
- Rust 第三方 MCP SDK（rmcp / mcp-sdk）API 仍在快速迭代（0.x），版本风险高
- 手撸 ~300 行代码不复杂，且零新依赖（仅 tokio io + serde_json，已有）
- 后续如需切换到 SDK，AppState trait object 设计允许平滑迁移

### 协议核心

**JSON-RPC 2.0 over stdio · 一行一消息**：

```
Client → {"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}
Server ← {"jsonrpc":"2.0","id":1,"result":{"protocolVersion":"2024-11-05","serverInfo":{...},"capabilities":{"tools":{}}}}

Client → {"jsonrpc":"2.0","method":"notifications/initialized"}     (notification · no response)

Client → {"jsonrpc":"2.0","id":2,"method":"tools/list"}
Server ← {"jsonrpc":"2.0","id":2,"result":{"tools":[...4 tools...]}}

Client → {"jsonrpc":"2.0","id":3,"method":"tools/call","params":{"name":"arkui_search_docs","arguments":{...}}}
Server ← {"jsonrpc":"2.0","id":3,"result":{"content":[{"type":"text","text":"..."}],"isError":false}}
```

### 4 个工具（方案 §4.2 决策 2 完整对齐）

| Tool | 状态 | 说明 |
|---|---|---|
| `arkui_search_docs` | ✅ 真活 | 检索文档 / API · 支持 expand_parent |
| `arkui_search_code` | ✅ 真活 | 检索代码示例 · 默认 expand_parent=true |
| `arkui_migrate_snippet` | ⏳ stub | 仅返回相关迁移规则（不调 LLM 真生成）· Week 4-5 续 |
| `arkui_validate_api` | ⏳ stub | API 版本检查器待 Week 4 续 |

每个工具有完整 `inputSchema` JSON Schema 声明 → Claude Code 可自动生成 prompt。

### handle_line 设计（核心）

```rust
pub async fn handle_line(state: &Arc<AppState>, line: &str) -> Option<String> {
    // 解析 → 路由 method → 返回 Some(response_json) 或 None（notification）
}
```

公开为 `pub async fn` 以便集成测直接调用，不需要起真 stdio 循环。

### 输出格式

`tools/call` 返回 `content: [{type: "text", text: "..."}]`，text 是 markdown：
- `# 📚 文档检索 · Top-K` 标题
- 每个 hit：`## [N] source · Lx-y · score`
- code block + heading_path + optional parent context

LLM 友好（含上下文行号 + 引用路径） + 人类可读。

### CLI 互斥校验

```
--http 占用 TCP 端口（127.0.0.1:7654）
--mcp 占用 stdio（stdin/stdout）
```

二者不可同时启用 → CLI 校验。LSP 仍 stub（Day 16 续）。

### 替代方案权衡（被否）

- 备选：用 `rmcp` 第三方 SDK
  - 否决：0.x 版本风险 · 协议本身简单 · 手撸更稳
- 备选：MCP SSE (HTTP+EventStream) 而非 stdio
  - 否决：stdio 是 Claude Code / Cursor 默认接入方式 · 方案 §4.4 明确 stdio + SSE 双协议；SSE 留 Day 15 续
- 备选：暴露 4 个独立工具 vs 一个通用 `arkui_query` 工具
  - 否决：MCP 设计哲学是"工具 name 即语义" · 4 个分明的 name 让 LLM 路由更准
- 备选：把 migrate / validate 真活（调 LLM）
  - 否决：Day 15 范围裁剪 · 需要 LLM 接入抽象 + 多模型适配（方案 §3.3）· Week 4-5 单独切片

## 改动要点

> API 选型 / 算法 / 关键决策

**与 Day 14 的差异**：
- crate 数 9（不变）
- 测试数 mcp feature 加 6 单测：initialize/tools_list/search_docs/unknown_method/notifications/migrate
- CLI feature 数 8 → 9（+ mcp）
- full feature 加 mcp → 一键全启 9 项
- AppState 类型在 lib.rs 抽出 alias，让 mcp-only 编译路径也能用

**API 选型**：
- `tokio::io::stdin/stdout` + `BufReader.lines()`（行级协议）
- `serde_json::Value` 灵活解析 JSON-RPC（不强 derive struct，协议层薄）
- `handle_line` 公开为 testable async fn（不起 stdio 循环）
- 错误码遵循 JSON-RPC 2.0 规范（-32700 parse / -32601 method not found / -32602 invalid params / -32603 internal）

**关键决策**：
- stdio 协议规避 stdout 污染（启动信息走 stderr · `eprintln!`）
- 工具输出 markdown 文本（LLM 友好）
- HTTP / MCP 共享 AppState trait object → CLI 复用 build_* helper
