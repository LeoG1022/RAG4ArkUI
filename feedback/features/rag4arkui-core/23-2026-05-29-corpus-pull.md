## 本轮目标

让用户拿到 release binary 后**不用手动投放 corpus 文档**：

```bash
arkui-rag corpus pull --target ./corpus/official
arkui-rag index --source ./corpus/official --bm25 tantivy
arkui-rag query --text "..."
```

完整支持：
- `--url <URL>`：HTTP 下载远端 tarball（默认指向 GitHub Releases）
- `--from-file <PATH>`：离线 / 测试场景 · 跳过 HTTP 读本地 tarball
- `--strip-components N`：兼容不同 tarball 顶层结构（默认 1，剥外层 wrap 目录）
- `--force`：覆盖非空目标目录
- Path traversal 安全检查（防恶意 tarball 写到 target 外）
- 500 MB 下载上限（防恶意 HTTP 响应吃光内存）

## Plan

> 必备节（FAIL 级 M-FEATURE-PLAN）

### HTTP 客户端选型

| 选项 | 体积 | TLS 依赖 | 同步/异步 | 选择 |
|---|---|---|---|---|
| `reqwest` | 大（拉 tokio + hyper + 一堆） | rustls / native-tls | async | ❌ 重 |
| `ureq` | 小（自包含 + rustls） | rustls 内置 | sync | ✅ 选 |
| `curl` crate | 中 | 系统 libcurl | sync | ❌ 引入系统库依赖 |

**结论**：`ureq` 同步 API + `tokio::task::spawn_blocking` 桥接。优势：
- 零 tokio runtime 依赖
- rustls 内置（无 OpenSSL / libcrypto 系统库）
- 体积小（vs reqwest）
- gzip 自动解压（response 层）
- 4 MB 体积增量（10.6 MB → 11 MB binary）

### Tarball 格式契约

```
arkui-rag-corpus-vX.Y.Z.tar.gz
└── arkui-rag-corpus-vX.Y.Z/      <-- strip_components=1 默认剥掉
    └── （任意子目录结构 · 通常是 markdown / 代码示例）
```

兼容性：用户的 tarball 顶层是 `arkui-rag-corpus-vX.Y.Z/` 单一目录 → 解到 `target/...` 不带 wrap 目录。

如果 tarball 内已经是裸文件（无 wrap 目录），用 `--strip-components=0`。

### 安全考虑

| risk | 缓解 |
|---|---|
| 恶意 tarball 含 `../../etc/passwd` 类 path traversal | 解压前 canonicalize 每个 out_path · 不在 target 内则 bail |
| HTTP 响应巨大吃光内存 | `Read::take(500 * 1024 * 1024)` 上限 500 MB |
| 目标目录非空被静默覆盖 | 默认拒绝 · 必须 `--force` 显式 |
| HTTP 超时悬挂 | `timeout(180s)` |

### Feature gate 策略

`corpus-pull` 作为可选 feature（默认 release 启用，但允许极小 binary 不带）：
- 单纯 server（HTTP/MCP/LSP 三协议） + 不需要 corpus pull → `cargo build --features http,mcp,lsp,tantivy` （8 MB）
- 全功能用户向 CLI → `cargo build --features full`（含 corpus-pull · 11 MB）

cli/src/main.rs 的 `Pull` arm 在 no-feature 下 bail：
```rust
#[cfg(not(feature = "corpus-pull"))]
anyhow::bail!("corpus pull 需要 corpus-pull feature 启用");
```

### 替代方案权衡（被否）

- 备选 1：用 `reqwest` async（与 tokio 配套）
  - 否决：体积 +5+ MB · 拉 hyper + 一堆 transitive deps · ureq 同步 + spawn_blocking 已足够
- 备选 2：用系统 `curl` shell out
  - 否决：跨平台脆弱（Windows 没 curl）· 进程 fork 开销 · stream 处理麻烦
- 备选 3：内嵌默认 corpus（`include_bytes!` 编译时打包）
  - 否决：binary 膨胀且每次升级 corpus 都要重 release · 网络下载更灵活
- 备选 4：把 corpus pull 实装到独立 crate `arkui-rag-corpus`
  - 否决：单一 CLI 子命令 · 不值得拆 crate · 110 行代码够轻

## 改动要点

> API 选型 / 算法 / 关键决策

### 与 Day 20c 的差异

- crate 数 9（不变）
- CLI 子命令：4 → 5（+ `corpus pull`）
- 默认 release features：5 → **6 项**（+ corpus-pull）
- binary 大小：6.7 MB → **11 MB**（+ ureq + tar + flate2 ~4 MB）
- tarball 大小：2.9 MB → **4.1 MB**（gzip 后增量约 1.2 MB）

### 关键决策

1. **ureq + spawn_blocking**：同步 HTTP 在异步 corpus_op 里包一层 spawn_blocking · 不引入 tokio runtime 入侵
2. **strip_components 默认 1**：约定 tarball 顶层是 wrap 目录 · 跟 cargo / npm 等 release 包惯例对齐
3. **Path traversal 检查 + 500 MB 上限**：诚实考虑攻击面 · 即使 corpus pull 是开发者工具不是 server
4. **`--from-file` 路径**：CI / 离线 / 测试场景关键 · 实现成本几乎 0（已经要解压 bytes，只是来源换成 file）
