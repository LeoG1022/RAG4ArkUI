## Plan

> 必备节（FAIL 级 M-FEATURE-PLAN）

### 设计：手撸 LSP framing（不引入 tower-lsp）

**理由**（与 mcp.rs 决策一致）：
- LSP 与 MCP 都是 JSON-RPC 2.0，骨架可复用
- LSP 仅多一层 **Content-Length** 帧封装（vs MCP 行分隔）
- tower-lsp 0.x 版本风险高 + 引入大量 type-state 抽象
- 手撸 framing ~50 行 · 复用 mcp.rs 的 JSON-RPC 处理风格

### LSP 协议核心

```
Client → Content-Length: <N>\r\n\r\n<json body>
         {"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}

Server ← Content-Length: <N>\r\n\r\n<json body>
         {"jsonrpc":"2.0","id":1,"result":{"capabilities":{...},"serverInfo":{...}}}
```

### 实装 method（5 真活 + 2 stub + 5 忽略）

| Method | 状态 | 说明 |
|---|---|---|
| `initialize` | ✅ 真活 | 返回 capabilities · hoverProvider + executeCommandProvider |
| `initialized` | ✅ notification | 无响应 |
| `shutdown` | ✅ 真活 | 返回 null · 设 shutdown_requested |
| `exit` | ✅ notification | 停 server（is_exit=true） |
| `arkui-rag/search` | ✅ 真活 | 自定义 method · 返回 markdown content + hits |
| `arkui-rag/migrate` | ⏳ stub | 返回提示 · Week 4-5 接 LLM |
| `textDocument/hover` | ⏳ stub | capability=true 但返回 null · Day 16 续真活 |
| `textDocument/didOpen/didChange/didClose` | ✅ 已知 notification · 忽略 |
| `$/cancelRequest`, `$/setTrace` | ✅ 已知 notification · 忽略 |

shutdown 状态机：shutdown 后只允许 exit · 其他 method 返回 error（LSP 规范）。

### 与 mcp.rs 的差异

| 维度 | mcp.rs | lsp.rs |
|---|---|---|
| Framing | 行分隔（`\n`） | `Content-Length: N\r\n\r\n` |
| State machine | 无 | shutdown_requested 状态 |
| Custom method namespace | 工具调用走 `tools/call` | 自定义 method 直接 `arkui-rag/*` |
| Capabilities | 仅 tools | tools + hover + executeCommand |
| Exit | stdin EOF | `exit` notification 显式停 |

### CLI 三协议互斥

```rust
let selected = (http as u8) + (mcp as u8) + (lsp as u8);
if selected == 0 { bail!("必须指定一个协议") }
if selected > 1 { bail!("--http / --mcp / --lsp 三者互斥") }
```

stdio 协议同进程只能占一次（stdin/stdout）。如需多协议同时跑，启多个进程。
